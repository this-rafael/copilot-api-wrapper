# Copilot API Wrapper V2 — Plano de Migração de Stack

> **Stack proposta:** Backend Go · Frontend Next.js · Mobile React Native · MCP TypeScript
>
> Plano criado em 15/03/2026 com validações na internet (pkg.go.dev, npm, GitHub, StackOverflow, context7).

---

## Sumário Executivo

Migrar o copilot-api-wrapper de uma stack 100% Node.js/TypeScript para uma arquitetura poliglota:

| Camada | V1 (atual) | V2 (proposta) |
|---|---|---|
| **Backend** | Node.js + TypeScript + `ws` + `node-pty` | **Go** + gorilla/websocket + creack/pty |
| **Frontend Web** | React 19 + Vite 6 + xterm.js (SPA puro) | **Next.js 16** + React 19 + xterm.js (App Router) |
| **Mobile** | Responsive web (PWA-like) | **React Native** + Expo + WebView + xterm.js |
| **MCP Server** | TypeScript + Express + Streamable HTTP  | **TypeScript** (mantido — MCP SDK oficial é TS) |
| **Autocomplete/LSP** | Node.js JSON-RPC client sobre stdio | **Go** JSON-RPC 2.0 client sobre stdio |

## Parte 2 — Validação das Tecnologias Propostas

### 2.1 Backend Go: WebSocket + PTY

#### gorilla/websocket ✅ VALIDADO
- **Fonte:** pkg.go.dev, context7 (28 snippets, reputation: High, score: 73.7)
- **Capacidades confirmadas:**
  - `websocket.Upgrader` com `CheckOrigin` customizável
  - `ReadMessage()` / `WriteMessage()` loop-based
  - Ping/Pong handlers com `SetPongHandler()` + deadline renewal
  - Compression: per-message deflate nativo
  - Concorrência: goroutine-safe para reads e writes separados
- **Alternativa avaliada:** `coder/websocket` (score: 84, mais idiomático com `context.Context`)
  - Reddit (r/golang, abr/2025): "I switched to coder/websocket when gorilla was originally deprecated and like it better"
- **Recomendação:** Usar `coder/websocket` (nhooyr) como primary — melhor suporte a context, API mais moderna

#### creack/pty ✅ VALIDADO
- **Fonte:** pkg.go.dev, context7 (25 snippets, reputation: High, score: 80.65)
- **Capacidades confirmadas:**
  - `pty.StartWithSize(cmd, &pty.Winsize{Rows, Cols})` — equivale ao `node-pty`
  - `pty.Setsize(ptmx, &pty.Winsize{})` — resize dinâmico
  - `pty.InheritSize(src, dst)` — propagação de SIGWINCH
  - Read/Write via standard `io.Reader` / `io.Writer`
  - Suporte Linux/macOS (não Windows — mesmo limite do node-pty)
- **Referência:** GoTTY (yudai/gotty) — terminal-over-web em Go usando creack/pty + gorilla/websocket
  - Projeto maduro com 18k+ stars no GitHub
  - Prova de conceito que Go + PTY + WebSocket funciona em produção

#### Go JSON-RPC 2.0 para LSP ✅ VALIDADO
- **Bibliotecas disponíveis:**
  - `sourcegraph/jsonrpc2` — client/server JSON-RPC 2.0
  - `owenrumney/go-lsp` — helper LSP 3.17+ com type definitions
  - Go `net/rpc/jsonrpc` — stdlib (v1 only, insuficiente para LSP)
  - Escrita manual: Content-Length framing é simples em Go (bufio.Reader + json.Decoder)
- **Recomendação:** Implementar JSON-RPC client custom (como o `JsonRpcProcess.ts` atual) — o protocolo é simples e o controle total é necessário para Copilot LSP
- **Validação StackOverflow:** "GitHub released their official language server for Copilot recently" — o subprocess spawning é a mesma técnica em Go que em Node

#### Vantagens do Go para este backend

| Aspecto | Node.js (V1) | Go (V2) |
|---|---|---|
| **Concorrência** | Event loop single-thread + worker threads | Goroutines nativas (M:N scheduler) |
| **Memória** | ~50-100MB base (V8 heap) | ~5-15MB base |
| **Binary** | Requer Node.js runtime instalado | Single static binary |
| **PTY performance** | `node-pty` N-API bindings (overhead FFI) | `creack/pty` Go nativo (syscalls diretos) |
| **Deploy** | `node_modules` + runtime | 1 binário estático (~15-20MB) |
| **Startup** | ~500ms-1s | ~10-50ms |

#### Riscos identificados

| Risco | Impacto | Mitigação |
|---|---|---|
| Copilot LSP depende de Node.js | Alto | Go backend spawna `node` como subprocess — mesmo que V1 |
| `creack/pty` não suporta Windows | Médio | Já era limitação do `node-pty` na V1; usar ConPTY wrapper no futuro |
| Perda da tipagem unificada TS frontend↔backend | Médio | Gerar tipos TS do protocolo a partir de Go structs (code generation) |

### 2.2 Frontend Next.js: xterm.js + App Router

#### Compatibilidade xterm.js ✅ VALIDADO COM RESSALVAS
- **Problema conhecido:** xterm.js depende de DOM APIs (`window`, `document`, `self`)
  - GitHub Discussion #22409 (vercel/next.js): "Xterm.js not work with NextJs"
  - GitHub Discussion #56241: solucionado com dynamic import
  - StackOverflow: "self is not defined" ao importar xterm
- **Solução confirmada (Next.js docs oficiais):**
  ```tsx
  'use client'
  import dynamic from 'next/dynamic'
  const TerminalView = dynamic(() => import('./TerminalView'), { ssr: false })
  ```
- **StackOverflow (mar/2024):** "XtermJS cannot read properties of undefined (reading dimensions) v5.3 on Next.js" — resolvido com `ssr: false`

#### Vantagens do Next.js para este projeto

| Capacidade | Benefício |
|---|---|
| **App Router** | Layouts aninhados: `/connection` e `/terminal` como rotas |
| **Server Components** | ConnectionScreen pode ter validação server-side |
| **API Routes** | REST endpoints auxiliares sem servidor separado |
| **Static Export** | `output: 'export'` para deploy como SPA estático se necessário |
| **Image Optimization** | Screenshots e assets otimizados |
| **Middleware** | Validação de auth antes do WebSocket upgrade |
| **Code splitting** | Terminal + addons carregados em lazy via dynamic() |

### 2.3 Mobile React Native: Terminal Emulation

#### @fressh/react-native-xtermjs-webview ✅ VALIDADO
- **NPM:** `@fressh/react-native-xtermjs-webview` (publicado Oct 2025)
- **Descrição:** "React Native WebView that embeds xterm.js with sensible defaults and a bridge for input and output"
- **Abordagem:** WebView renderiza xterm.js; bridge `postMessage` para input/output
- **Resultado da pesquisa Google:** Encontrado como resultado #2 para "react native webview xterm.js terminal"

#### Estratégia de Terminal no React Native

**Opção A — WebView com xterm.js (RECOMENDADO)**
- `react-native-webview` renderiza uma página HTML com xterm.js
- JavaScript bridge: `postMessage()` para enviar inputs, `onMessage` para receber outputs
- Reutiliza toda a lógica de rendering do frontend web
- Pacote `@fressh/react-native-xtermjs-webview` encapsula isso

**Opção B — Terminal Canvas nativo**
- Renderizar glyphs em `react-native-skia` ou Canvas nativo
- Performance superior, mas esforço de desenvolvimento enorme
- Não vale a pena para este escopo

**Opção C — Shared codebase (Monorepo)**
- Expo com `react-native-web` compila React Native → web
- Porém, xterm.js é incompatível com RN nativo
- Melhor abordagem: shared hooks + WebView terminal

### 2.4 MCP Server: Manter em TypeScript ✅ VALIDADO

#### Justificativa para NÃO migrar o MCP

| Fator | Decisão |
|---|---|
| SDK oficial MCP é TypeScript | `@modelcontextprotocol/sdk` — 272 snippets, High reputation, score: 78.26 |
| MCP SDK Go existe mas é beta | `mcp-go` existe porém sem paridade de features |
| Escopo isolado | MCP server é independente — não compartilha código com backend |
| Já funciona | Servidor estável com state management, scheduler, firewall ops |
| Windows-specific | Lógica de `netsh` é shell scripting que TS handles bem |

#### Melhorias sugeridas no MCP V2

- Atualizar para último `@modelcontextprotocol/sdk` (checar breaking changes)
- Adicionar tool `diagnose-connectivity` para debug de rede
- Considerar MCP Streamable HTTP como transport padrão (já implementado)

## Parte 5 — Protocolo: Compatibilidade V1 ↔ V2

### 5.1 Protocolo WebSocket: MANTER 100% COMPATÍVEL

O protocolo WebSocket (17 tipos de mensagem) **NÃO MUDA**. Isso permite:

- Clients V1 (Vite) conectando em backend V2 (Go) durante transição
- Clients V2 (Next.js) conectando em backend V1 (Node) como fallback
- Testes de integração existentes continuam válidos

### 5.2 Geração de Tipos TypeScript a partir de Go

Para manter type-safety entre backend Go e frontends TS:

```go
// backend/internal/protocol/messages.go
type SessionCreateMessage struct {
    Type           string `json:"type"`
    CWD            string `json:"cwd"`
    CommandProfile string `json:"commandProfile"`
    Cols           int    `json:"cols"`
    Rows           int    `json:"rows"`
}
```

Opções:
1. **Manual:** Manter `protocol.ts` em sync manualmente (como V1)
2. **Code generation:** `tygo` (Go structs → TypeScript interfaces)
3. **JSON Schema:** Gerar schema do Go, validar com Zod no client

**Recomendação:** Usar `tygo` para gerar `packages/shared/src/protocol.generated.ts`

## Parte 6 — Decisões Técnicas Detalhadas

### 6.1 Go: Escolhas de Bibliotecas

| Necessidade | Biblioteca | Justificativa |
|---|---|---|
| WebSocket | `nhooyr.io/websocket` (coder/websocket) | Context-aware, API moderna, score 84 |
| PTY | `github.com/creack/pty/v2` | Único PTY lib maduro em Go, score 80.65 |
| SQLite | `github.com/mattn/go-sqlite3` ou `modernc.org/sqlite` | go-sqlite3 requer CGO; modernc.org é pure Go |
| Logging | `log/slog` (stdlib) | Estruturado, zero dependências, Go 1.21+ |
| Config | `github.com/caarlos0/env/v11` | Parsing env vars com struct tags |
| Validation | Custom ou `github.com/go-playground/validator/v10` | Validação de structs |
| JSON | `encoding/json` (stdlib) | Suficiente para o protocolo |
| UUID/ID | `github.com/rs/xid` | Globalmente único, sortable, 12 bytes |

### 6.2 Go: SQLite — CGO vs Pure Go

| Opção | Prós | Contras |
|---|---|---|
| `mattn/go-sqlite3` (CGO) | Performance máxima, feature-complete | Requer gcc, cross-compile difícil |
| `modernc.org/sqlite` (Pure Go) | Zero CGO, cross-compile fácil | ~2-3x mais lento, binário maior |
| `go.etcd.io/bbolt` (BoltDB) | Key-value puro Go, sem SQL | Sem queries SQL — mudança de paradigma |

**Recomendação:** `modernc.org/sqlite` — o volume de dados (custom workspaces) é minúsculo e o cross-compile fácil vale mais

| Aspecto | Decisão |

## Parte 10 — Riscos e Mitigações

| # | Risco | Probabilidade | Impacto | Mitigação |
|---|---|---|---|---|
| 1 | Copilot LSP requer Node.js runtime | Certa | Baixo | Go spawna `node` como subprocess — irrelevante para o rest do backend |
| 2 | xterm.js SSR crash no Next.js | Certa | Baixo | `dynamic(() => import(...), { ssr: false })` — solução documentada e validada |
| 3 | Performance do terminal em WebView (React Native) | Média | Médio | Benchmarkar; fallback para canvas nativo se necessário |
| 4 | `creack/pty` sem suporte Windows | Certa | Baixo | Já era limitação em V1; ConPTY via Windows API no futuro |
| 5 | Perda de type-safety TS entre Go e frontends | Média | Médio | Code generation com `tygo`; contract tests no CI |
| 6 | Complexidade do monorepo | Baixa | Médio | Turborepo + pnpm workspaces são battle-tested |
| 7 | Overhead de manutenção (Go + TS + RN) | Alta | Alto | Shared code maximizado; protocolo estável minimiza changes |
| 8 | SQLite pure Go (modernc.org) — performance | Baixa | Baixo | Volume de dados é minúsculo (~100 rows no máximo) |

## Parte 11 — Decisão: Vale a Pena?

### Argumentos A FAVOR da migração

1. **Single binary deploy** — Go backend é um executável estático sem dependências
2. **Performance massiva** — goroutines > event loop para múltiplas sessões PTY concorrentes
3. **Mobile nativo** — React Native permite distribuição via App Store/Play Store
4. **SEO/SSR** — Next.js permite landing page com SSR (ConnectionScreen)
5. **Menor footprint** — Go usa ~5-15MB de RAM vs ~50-100MB do Node.js
6. **Type-safe backend** — Go tem compilação estática que pega erros no build time

### Argumentos CONTRA

1. **Copilot LSP ainda requer Node.js** — não elimina a dependency de Node completamente
2. **Aumento de complexidade** — 3 linguagens (Go + TS + TSX) vs 1 (TS)
3. **Curva de aprendizado** — Go para quem vem de TS pode ser íngreme
4. **Este projeto é relativamente pequeno** — a sobrecarga de uma stack poliglota pode não compensar
5. **Manutenção dobrada** — protocolo precisa ser mantido em Go AND TypeScript

### Veredicto

A migração faz sentido SE:
- Há planos de escalar para muitas sessões concorrentes (>50)
- O mobile app é um requisito de negócio (não apenas nice-to-have)
- A equipe tem experiência com Go

Se o objetivo é apenas modernizar, considere **manter Node.js no backend** e migrar apenas o frontend para Next.js + adicionar React Native.
